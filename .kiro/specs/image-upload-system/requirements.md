# Requirements Document: Image Upload System

## Introduction

The Image Upload System enables administrators to securely upload, manage, and organize car images for Prime Deal Auto's inventory. The system uses presigned S3 URLs for direct browser-to-S3 uploads, eliminating the need to proxy large files through the Lambda backend. Images are stored in S3 with public read access (CloudFront integration deferred due to account restrictions), and metadata is tracked in Aurora PostgreSQL.

This feature is critical for the car listing workflow, as high-quality images are essential for customer engagement and conversion.

## Glossary

- **Image_Upload_System**: The complete backend and frontend system for uploading, storing, and managing car images
- **Presigned_URL**: A time-limited URL generated by AWS S3 that grants temporary upload permissions without requiring AWS credentials
- **S3_Bucket**: AWS Simple Storage Service bucket that stores car image files
- **Car_Images_Table**: PostgreSQL table storing image metadata (S3 keys, URLs, display order, primary flag)
- **Image_Uploader_Component**: React component providing drag-and-drop interface for image management
- **Admin_User**: Authenticated user with admin role in Cognito user pool
- **Primary_Image**: The main image displayed in car listings and search results
- **Display_Order**: Integer field determining the sequence in which images appear in the gallery
- **S3_Key**: The unique path identifier for an image file in S3 (format: `cars/{carId}/{timestamp}-{filename}`)
- **Image_Metadata**: Database record containing S3 key, URL, display order, primary flag, and timestamps
- **Upload_Progress**: Real-time feedback showing percentage completion of file upload
- **Image_Gallery**: Frontend component displaying car images in a browsable interface

## Requirements

### Requirement 1: Presigned URL Generation

**User Story:** As an admin user, I want to receive secure upload URLs, so that I can upload images directly to S3 without exposing AWS credentials.

#### Acceptance Criteria

1. WHEN an Admin_User requests an upload URL for a valid car, THE Image_Upload_System SHALL generate a Presigned_URL with 5-minute expiration
2. THE Image_Upload_System SHALL include the S3_Key in the Presigned_URL response
3. THE Image_Upload_System SHALL validate that the car exists before generating a Presigned_URL
4. THE Image_Upload_System SHALL validate that the car has fewer than 20 images before generating a Presigned_URL
5. THE Image_Upload_System SHALL return error code NOT_FOUND when the car does not exist
6. THE Image_Upload_System SHALL return error code VALIDATION_ERROR when the car already has 20 images
7. THE Image_Upload_System SHALL return error code UNAUTHORIZED when the user is not an Admin_User
8. THE Presigned_URL SHALL allow PUT requests with Content-Type of image/jpeg, image/png, or image/webp
9. THE S3_Key SHALL follow the format `cars/{carId}/{timestamp}-{filename}`
10. FOR ALL valid requests, generating a Presigned_URL SHALL NOT modify any database state

### Requirement 2: Image Metadata Storage

**User Story:** As an admin user, I want image metadata saved to the database after successful upload, so that images appear in the car listing.

#### Acceptance Criteria

1. WHEN an Admin_User confirms successful S3 upload, THE Image_Upload_System SHALL create an Image_Metadata record in the Car_Images_Table
2. THE Image_Upload_System SHALL store the S3_Key, S3 URL, car ID, and creation timestamp
3. WHEN the car has no existing images, THE Image_Upload_System SHALL set the new image as the Primary_Image
4. WHEN the car has existing images, THE Image_Upload_System SHALL set Display_Order to one greater than the current maximum
5. THE Image_Upload_System SHALL validate that the S3_Key matches the expected format before saving
6. THE Image_Upload_System SHALL return error code VALIDATION_ERROR when the S3_Key format is invalid
7. THE Image_Upload_System SHALL return error code UNAUTHORIZED when the user is not an Admin_User
8. THE Image_Upload_System SHALL return the complete Image_Metadata record including the generated ID
9. FOR ALL successful saves, the Image_Metadata record SHALL be immediately queryable by car ID

### Requirement 3: Image Deletion

**User Story:** As an admin user, I want to delete images I no longer need, so that I can maintain accurate car listings.

#### Acceptance Criteria

1. WHEN an Admin_User deletes an image, THE Image_Upload_System SHALL remove the file from the S3_Bucket
2. WHEN an Admin_User deletes an image, THE Image_Upload_System SHALL remove the Image_Metadata record from the Car_Images_Table
3. IF the deleted image was the Primary_Image AND other images exist, THEN THE Image_Upload_System SHALL promote the image with the lowest Display_Order to Primary_Image
4. THE Image_Upload_System SHALL return error code NOT_FOUND when the image does not exist
5. THE Image_Upload_System SHALL return error code UNAUTHORIZED when the user is not an Admin_User
6. THE Image_Upload_System SHALL return error code FORBIDDEN when the user does not own the car (future multi-dealer support)
7. THE Image_Upload_System SHALL complete successfully even if the S3 file does not exist (idempotent deletion)
8. FOR ALL successful deletions, the image SHALL NOT appear in subsequent car queries

### Requirement 4: Primary Image Selection

**User Story:** As an admin user, I want to set a primary image for each car, so that the best photo appears in search results and listings.

#### Acceptance Criteria

1. WHEN an Admin_User sets an image as primary, THE Image_Upload_System SHALL update the is_primary flag to true for the selected image
2. WHEN an Admin_User sets an image as primary, THE Image_Upload_System SHALL update the is_primary flag to false for all other images of the same car
3. THE Image_Upload_System SHALL validate that the image belongs to the specified car
4. THE Image_Upload_System SHALL return error code NOT_FOUND when the image does not exist
5. THE Image_Upload_System SHALL return error code VALIDATION_ERROR when the image does not belong to the specified car
6. THE Image_Upload_System SHALL return error code UNAUTHORIZED when the user is not an Admin_User
7. FOR ALL cars with images, exactly one image SHALL have is_primary set to true
8. FOR ALL successful primary image updates, the new primary image SHALL appear first in the Image_Gallery

### Requirement 5: Image Reordering

**User Story:** As an admin user, I want to reorder images by dragging them, so that I can control the sequence customers see in the gallery.

#### Acceptance Criteria

1. WHEN an Admin_User submits a new display order, THE Image_Upload_System SHALL update the Display_Order field for all affected images
2. THE Image_Upload_System SHALL validate that all image IDs in the request belong to the specified car
3. THE Image_Upload_System SHALL validate that Display_Order values are sequential integers starting from 0
4. THE Image_Upload_System SHALL return error code VALIDATION_ERROR when image IDs do not match the car
5. THE Image_Upload_System SHALL return error code VALIDATION_ERROR when Display_Order values are not sequential
6. THE Image_Upload_System SHALL return error code UNAUTHORIZED when the user is not an Admin_User
7. THE Image_Upload_System SHALL execute the reorder operation atomically within a database transaction
8. FOR ALL successful reorders, images SHALL appear in the Image_Gallery in the new Display_Order sequence

### Requirement 6: File Validation

**User Story:** As a system administrator, I want to enforce file type and size limits, so that only valid images are uploaded and storage costs are controlled.

#### Acceptance Criteria

1. THE Image_Upload_System SHALL accept only files with MIME types image/jpeg, image/png, or image/webp
2. THE Image_Upload_System SHALL reject files larger than 5 MB
3. THE Image_Upload_System SHALL validate file type on the frontend before requesting a Presigned_URL
4. THE Image_Upload_System SHALL validate file type in the Presigned_URL Content-Type restriction
5. THE Image_Upload_System SHALL return error code VALIDATION_ERROR when file type is not supported
6. THE Image_Upload_System SHALL return error code VALIDATION_ERROR when file size exceeds 5 MB
7. THE Image_Upload_System SHALL display a user-friendly error message for each validation failure
8. FOR ALL uploaded files, the Content-Type header SHALL match one of the allowed MIME types

### Requirement 7: Upload Progress Tracking

**User Story:** As an admin user, I want to see upload progress for each image, so that I know when uploads are complete and can identify failures.

#### Acceptance Criteria

1. WHEN an image upload is in progress, THE Image_Uploader_Component SHALL display a progress bar showing percentage completion
2. THE Image_Uploader_Component SHALL update the progress bar at least every 500 milliseconds during upload
3. WHEN an upload completes successfully, THE Image_Uploader_Component SHALL display a success indicator
4. WHEN an upload fails, THE Image_Uploader_Component SHALL display an error message with the failure reason
5. THE Image_Uploader_Component SHALL allow cancellation of in-progress uploads
6. WHEN a user cancels an upload, THE Image_Uploader_Component SHALL abort the S3 request and remove the progress indicator
7. THE Image_Uploader_Component SHALL support simultaneous upload of multiple images with independent progress tracking
8. FOR ALL uploads, the progress percentage SHALL be calculated from bytes uploaded divided by total file size

### Requirement 8: Image Preview

**User Story:** As an admin user, I want to see thumbnail previews of images before and after upload, so that I can verify I'm uploading the correct files.

#### Acceptance Criteria

1. WHEN a user selects files for upload, THE Image_Uploader_Component SHALL display thumbnail previews using browser FileReader API
2. THE Image_Uploader_Component SHALL display thumbnails at 150x150 pixels with object-fit cover
3. WHEN an upload completes, THE Image_Uploader_Component SHALL replace the local preview with the S3 URL thumbnail
4. THE Image_Uploader_Component SHALL display the Primary_Image with a visual indicator (badge or border)
5. THE Image_Uploader_Component SHALL display Display_Order numbers on each thumbnail
6. THE Image_Uploader_Component SHALL support drag-and-drop reordering of thumbnails
7. WHEN a user drags a thumbnail to a new position, THE Image_Uploader_Component SHALL update the visual order immediately
8. FOR ALL thumbnail displays, images SHALL maintain aspect ratio without distortion

### Requirement 9: S3 Bucket Configuration

**User Story:** As a system administrator, I want S3 configured for public read access and automatic cleanup, so that images are accessible to customers and incomplete uploads don't accumulate.

#### Acceptance Criteria

1. THE S3_Bucket SHALL allow public read access on all objects in the `cars/` prefix
2. THE S3_Bucket SHALL have a lifecycle policy that deletes incomplete multipart uploads after 1 day
3. THE S3_Bucket SHALL use the folder structure `cars/{carId}/{timestamp}-{filename}` for all image keys
4. THE S3_Bucket SHALL have versioning disabled to minimize storage costs
5. THE S3_Bucket SHALL have server-side encryption enabled (AES-256)
6. THE S3_Bucket SHALL block public write access (only presigned URLs allow uploads)
7. THE S3_Bucket SHALL have CORS configured to allow PUT requests from the frontend domain
8. FOR ALL uploaded images, the S3 URL SHALL be publicly accessible without authentication

### Requirement 10: Database Schema Integrity

**User Story:** As a database administrator, I want referential integrity enforced, so that orphaned image records cannot exist.

#### Acceptance Criteria

1. THE Car_Images_Table SHALL have a foreign key constraint on car_id referencing the cars table
2. WHEN a car is deleted, THE Image_Upload_System SHALL cascade delete all associated Image_Metadata records
3. THE Car_Images_Table SHALL have a unique constraint on (car_id, display_order) to prevent duplicate ordering
4. THE Car_Images_Table SHALL have a check constraint ensuring Display_Order is non-negative
5. THE Car_Images_Table SHALL have a check constraint ensuring at most one image per car has is_primary set to true
6. THE Car_Images_Table SHALL have an index on car_id for efficient queries
7. THE Car_Images_Table SHALL have an index on (car_id, display_order) for efficient ordering queries
8. FOR ALL Image_Metadata records, the car_id SHALL reference an existing car in the cars table

### Requirement 11: Concurrent Upload Handling

**User Story:** As an admin user, I want to upload multiple images simultaneously, so that I can efficiently add photos to new car listings.

#### Acceptance Criteria

1. THE Image_Uploader_Component SHALL support selection of up to 20 files at once
2. THE Image_Uploader_Component SHALL initiate uploads in parallel with a maximum of 3 concurrent requests
3. WHEN multiple uploads are in progress, THE Image_Uploader_Component SHALL display individual progress for each file
4. THE Image_Uploader_Component SHALL continue processing remaining uploads when one upload fails
5. THE Image_Uploader_Component SHALL display a summary showing successful and failed upload counts
6. THE Image_Upload_System SHALL handle concurrent metadata save requests without race conditions
7. THE Image_Upload_System SHALL use database transactions to ensure Display_Order values remain sequential
8. FOR ALL concurrent uploads, the final Display_Order values SHALL be sequential without gaps

### Requirement 12: Error Recovery

**User Story:** As an admin user, I want clear error messages and retry options, so that I can resolve upload failures without losing progress.

#### Acceptance Criteria

1. WHEN a Presigned_URL expires before upload completes, THE Image_Uploader_Component SHALL display an error message indicating expiration
2. WHEN an S3 upload fails due to network error, THE Image_Uploader_Component SHALL offer a retry button
3. WHEN metadata save fails after successful S3 upload, THE Image_Upload_System SHALL log the orphaned S3_Key for cleanup
4. THE Image_Uploader_Component SHALL display specific error messages for: file too large, invalid file type, network error, server error, permission denied
5. THE Image_Uploader_Component SHALL allow users to remove failed uploads from the queue without affecting successful uploads
6. THE Image_Upload_System SHALL return structured error responses with error codes and human-readable messages
7. THE Image_Upload_System SHALL log all errors to CloudWatch with request context for debugging
8. FOR ALL error scenarios, the user SHALL receive actionable feedback on how to resolve the issue

### Requirement 13: Integration with Car Detail Page

**User Story:** As a customer, I want to see car images in a gallery on the detail page, so that I can view all available photos of the car.

#### Acceptance Criteria

1. WHEN a customer views a car detail page, THE Image_Gallery SHALL display all images ordered by Display_Order
2. THE Image_Gallery SHALL display the Primary_Image first regardless of Display_Order
3. THE Image_Gallery SHALL support navigation between images using previous/next buttons
4. THE Image_Gallery SHALL support keyboard navigation (arrow keys)
5. THE Image_Gallery SHALL display a thumbnail strip showing all available images
6. WHEN a customer clicks a thumbnail, THE Image_Gallery SHALL display the full-size image
7. THE Image_Gallery SHALL use lazy loading for images not currently visible
8. FOR ALL car detail pages with images, the Primary_Image SHALL be visible without scrolling

### Requirement 14: Admin Car Form Integration

**User Story:** As an admin user, I want the image uploader integrated into the car creation/edit form, so that I can manage images alongside other car details.

#### Acceptance Criteria

1. WHEN an Admin_User creates a new car, THE admin form SHALL display the Image_Uploader_Component
2. WHEN an Admin_User edits an existing car, THE Image_Uploader_Component SHALL load and display existing images
3. THE Image_Uploader_Component SHALL disable uploads until the car is saved to the database
4. THE admin form SHALL display a warning when attempting to save a car without a Primary_Image
5. THE admin form SHALL allow saving a car without images (images can be added later)
6. THE Image_Uploader_Component SHALL refresh the image list after successful uploads
7. THE Image_Uploader_Component SHALL use TanStack Query mutations for all API calls
8. FOR ALL car edit operations, image changes SHALL be saved independently of other car field updates

### Requirement 15: Performance Optimization

**User Story:** As a system administrator, I want efficient image loading and minimal Lambda execution time, so that costs are controlled and user experience is fast.

#### Acceptance Criteria

1. THE Image_Upload_System SHALL generate Presigned_URLs without loading image data into Lambda memory
2. THE Image_Upload_System SHALL use database connection pooling to minimize connection overhead
3. THE Image_Uploader_Component SHALL compress images client-side before upload when original size exceeds 3 MB
4. THE Image_Uploader_Component SHALL use browser-native image compression APIs when available
5. THE Image_Gallery SHALL use responsive image sizing to load appropriate resolutions for device screen size
6. THE Image_Gallery SHALL implement intersection observer for lazy loading off-screen images
7. THE Image_Upload_System SHALL complete Presigned_URL generation in under 200 milliseconds
8. FOR ALL image operations, Lambda execution time SHALL be under 1 second excluding S3 upload time

## Special Requirements Guidance

### Parser and Serializer Requirements

This feature does not include parsers or serializers for structured data formats. Image metadata is stored as standard PostgreSQL columns, and API requests/responses use JSON (handled by the existing API framework).

### Round-Trip Properties

The following round-trip properties should be tested:

1. **Image Upload Round-Trip**: Upload an image → retrieve car images → verify the uploaded image appears with correct metadata
2. **Reorder Round-Trip**: Set display order → retrieve car images → verify images appear in the specified order
3. **Primary Image Round-Trip**: Set image as primary → retrieve car images → verify exactly one image has is_primary=true and it matches the selected image
4. **Delete and Restore Round-Trip**: Delete an image → verify it's gone → upload a new image with the same display order → verify the new image appears correctly

### Invariant Properties

1. **Primary Image Invariant**: For all cars with at least one image, exactly one image SHALL have is_primary=true
2. **Display Order Invariant**: For all cars, Display_Order values SHALL be sequential integers starting from 0 with no gaps
3. **Image Count Invariant**: For all cars, the count of Image_Metadata records SHALL be less than or equal to 20
4. **Foreign Key Invariant**: For all Image_Metadata records, the car_id SHALL reference an existing car
5. **S3 Key Format Invariant**: For all Image_Metadata records, the S3_Key SHALL match the pattern `cars/{uuid}/{timestamp}-{filename}`

### Idempotence Properties

1. **Delete Idempotence**: Deleting an image multiple times SHALL produce the same result (image is gone, no error)
2. **Set Primary Idempotence**: Setting the same image as primary multiple times SHALL produce the same result (that image is primary, others are not)
3. **Metadata Save Idempotence**: Saving metadata for the same S3_Key multiple times SHALL update the existing record, not create duplicates

### Error Condition Properties

1. **Invalid File Type**: Uploading a file with MIME type application/pdf SHALL return VALIDATION_ERROR
2. **Oversized File**: Uploading a file larger than 5 MB SHALL return VALIDATION_ERROR
3. **Expired Presigned URL**: Attempting to upload after 5 minutes SHALL fail with S3 AccessDenied error
4. **Non-existent Car**: Requesting upload URL for non-existent car SHALL return NOT_FOUND
5. **Image Limit Exceeded**: Requesting upload URL when car has 20 images SHALL return VALIDATION_ERROR
6. **Unauthorized Access**: Non-admin user attempting any image operation SHALL return UNAUTHORIZED
7. **Invalid Display Order**: Submitting reorder with non-sequential values SHALL return VALIDATION_ERROR
8. **Cross-Car Image Reference**: Setting primary image with imageId from different car SHALL return VALIDATION_ERROR
